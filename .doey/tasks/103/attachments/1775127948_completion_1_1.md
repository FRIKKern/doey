● Research Report: Plans Tab Mouse Lock (Task #103)

  Summary

  The Plans tab has a cardHeight mismatch that corrupts all Y-coordinate-based click
  detection, plus it silently consumes every mouse event it doesn't explicitly handle. The tab
   bar switching works correctly (root model handles it first), but within the Plans tab,
  mouse interactions are broken.

  ---
  Bug 1 (Primary): cardHeight mismatch — plans.go:74 vs plans.go:358

  The plan card delegate declares a height of 3 lines:
  // plans.go:74
  func (d planCardDelegate) Height() int { return 3 }

  But the click detection uses cardHeight = 2:
  // plans.go:358
  const cardHeight = 2

  Compared to Tasks — the task card delegate at tui/internal/taskcard/taskcard.go:53 returns
  Height() = 2, matching its cardHeight = 2 at tasks.go:449. Plans is the only tab with this
  mismatch.

  Effect: With 3-line cards but 2-line click math, index = firstVisible + relY/2:
  - Clicking the bottom third of card 0 selects card 1
  - Clicking the middle of card 1 selects card 2
  - Every click is shifted, and the detection region extends 50% past the actual cards

  Bug 2: headerLines = 1 may be wrong — plans.go:356

  // plans.go:356
  const headerLines = 1

  Tasks uses headerLines = 2 (tasks.go:450). If the Plans left panel has more than 1 header
  line (the "PLANS" header), this adds further Y offset error to every card click.

  Bug 3: Missing GotoTop() on card select — plans.go:365-367

  // plans.go:365-367
  m.list.Select(index)
  m.leftFocused = false
  m.loadSelectedDetail()

  Tasks calls m.detailViewport.GotoTop() before loadSelectedDetail() (tasks.go:460). Plans
  doesn't, so selecting a new plan keeps the previous plan's scroll position.

  Bug 4: All unmatched mouse events consumed — plans.go:392

  // plans.go:392
  return m, nil

  Any tea.MouseMsg that isn't a release-on-zone, release-on-card, or wheel event is silently
  consumed. This includes regular left-click presses, motion events, and right clicks. While
  Tasks has the same pattern (tasks.go:497), it doesn't have the cardHeight bug amplifying the
   problem.

  Not a bug: Tab bar switching

  The root model at root.go:322-329 checks tab zones before forwarding mouse events to
  sub-models. Tab clicks that land on a tab-N zone return immediately, never reaching the
  Plans model. This works correctly regardless of which tab is active.

  How the bugs combine into "mouse lock"

  1. User clicks a plan card → cardHeight mismatch selects the wrong card (or selects a card
  when clicking empty space)
  2. leftFocused becomes false → detail mode
  3. In detail mode, clicking the left panel triggers the same broken card math → selects
  wrong card again
  4. Clicking the right panel (not on a button zone) → consumed silently at line 392
  5. There is no mouse-based way to return to left-panel focus — only keyboard (h, Esc, Tab)
  6. Combined: clicks feel "trapped" because they either mis-select or do nothing
