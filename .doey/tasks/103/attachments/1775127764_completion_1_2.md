  The root model checks tab zones only on MouseActionRelease (line 322). This is correct. But
  the issue is more subtle:

  The zone.Get("tab-N").InBounds(msg) check uses absolute terminal coordinates. The tab bar
  zones are computed from the rendered output by zone.Scan(). If the coordinates returned by
  zone.Get() don't match where the tabs actually are on screen, tab clicks won't register.

  Looking at the flow more carefully — when the Plans tab is active and the user clicks where
  the tab bar is:

  1. Root checks zone.Get("tab-0").InBounds(msg) through zone.Get("tab-7").InBounds(msg) — if

  2. If none match (which shouldn't happen for a valid tab click), it forwards to
