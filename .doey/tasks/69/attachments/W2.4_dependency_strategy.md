# S7: Dependency Strategy — gum for Doey CLI

## 1. Current Install Flow Analysis

**install.sh** uses a tiered dependency model:

| Dependency | Type | Install Method | Fallback |
|------------|------|----------------|----------|
| git | Required | `brew install` / `apt install` | Dies if missing |
| tmux | Required | `brew install` / `apt install` | Dies if missing |
| Node.js | Soft required | `brew install` / `apt` / `fnm` | Warning only (needed for Claude CLI) |
| Claude CLI | Soft required | `npm install -g` | Warning only |
| jq | Optional | Not installed | `python3` fallback |
| Go | Optional | `brew install` | info-panel.sh fallback |
| doey-tui | Optional | Built from source (Go) | info-panel.sh fallback |

**Key patterns already in use:**
- `has()` function: `command -v "$1" &>/dev/null`
- `ask_install()` — interactive Y/n prompt for missing tools
- `require_tool()` — installs via brew (macOS) or apt (Linux)
- `run_install()` — wraps the install command with status output
- Platform detection: `uname -s` → `macos` / `linux`
- Homebrew detection: `HAS_BREW` flag

**web-install.sh** flow: `curl | bash` → clones repo to temp dir → runs `install.sh`. Only hard requirement is git.

## 2. gum Installation Options

### Binary Release Stats (v0.17.0)

| Platform | Archive | Size |
|----------|---------|------|
| Darwin arm64 | `.tar.gz` | 4.2 MB |
| Darwin x86_64 | `.tar.gz` | 4.6 MB |
| Linux x86_64 | `.tar.gz` | 4.5 MB |
| Linux arm64 | `.tar.gz` | 4.1 MB |

### Installation Methods

| Method | Pros | Cons | Best for |
|--------|------|------|----------|
| `brew install gum` | Simple, auto-updates | Requires Homebrew, slow (~30s) | macOS users with brew |
| Direct binary download | No dependencies, fast (~3s), exact version | Must detect arch, manual updates | `curl \| bash` installs |
| `go install` | From source | Requires Go 1.17+, compiles (~60s) | Developers |
| `apt install gum` | System package | Requires adding Charm repo first | Debian/Ubuntu |
| `pacman -S gum` | Native | Arch only | Arch Linux |

### Recommended: Direct binary download

For a `curl | bash` installer, the direct binary approach is ideal:

```bash
# Pattern: detect platform + arch → download → extract → install
GUM_VERSION="0.17.0"
ARCH=$(uname -m)
OS=$(uname -s)
case "$OS/$ARCH" in
  Darwin/arm64)  GUM_ARCHIVE="gum_${GUM_VERSION}_Darwin_arm64.tar.gz" ;;
  Darwin/x86_64) GUM_ARCHIVE="gum_${GUM_VERSION}_Darwin_x86_64.tar.gz" ;;
  Linux/x86_64)  GUM_ARCHIVE="gum_${GUM_VERSION}_Linux_x86_64.tar.gz" ;;
  Linux/aarch64) GUM_ARCHIVE="gum_${GUM_VERSION}_Linux_arm64.tar.gz" ;;
  *) GUM_ARCHIVE="" ;;  # Skip unsupported platforms
esac

if [ -n "$GUM_ARCHIVE" ]; then
  curl -fsSL "https://github.com/charmbracelet/gum/releases/download/v${GUM_VERSION}/${GUM_ARCHIVE}" \
    | tar xz -C ~/.local/bin gum
fi
```

- 4-5 MB download (comparable to Go binary for doey-tui)
- Takes ~3 seconds on broadband
- No build toolchain needed
- Extracts single binary — no dependencies

## 3. Fresh Install Constraint Analysis

CLAUDE.md mandates:
> "Would this work after deleting ~/.config/doey/, ~/.local/bin/doey, ~/.claude/agents/doey-* and running ./install.sh fresh?"

**gum fits cleanly because:**
- Single static binary → drops into `~/.local/bin/` (already in PATH for Doey users)
- No runtime dependencies (no shared libs, no config files)
- No user-level config needed
- Same install pattern as doey-tui (binary into `~/.local/bin/`)

**Risks:**
- Network dependency during install (GitHub Releases API)
- ~4.5 MB additional download time
- GitHub rate limits on unauthenticated API calls (but direct release URLs don't hit the API)

## 4. Graceful Fallback Strategy

### Design Principle: Enhancement, Not Requirement

gum should be a **visual enhancement layer** — every feature must work without it. The pattern is the same one Doey already uses for jq (`python3` fallback) and doey-tui (`info-panel.sh` fallback).

### Detection Pattern

```bash
# In common library (e.g., shell/doey-ui.sh or inline)
HAS_GUM=false
command -v gum &>/dev/null && HAS_GUM=true
```

### Wrapper Functions (Proposed)

Each interactive primitive gets a wrapper that dispatches to gum or plain bash:

```bash
# Confirmation prompt
doey_confirm() {
  local prompt="$1"
  if [ "$HAS_GUM" = true ]; then
    gum confirm "$prompt"
  else
    printf "  %s [y/N] " "$prompt"
    read -r reply
    [[ "$reply" =~ ^[Yy]$ ]]
  fi
}

# Text input
doey_input() {
  local prompt="$1" placeholder="${2:-}"
  if [ "$HAS_GUM" = true ]; then
    gum input --prompt "$prompt " --placeholder "$placeholder"
  else
    printf "  %s " "$prompt"
    read -r reply
    echo "$reply"
  fi
}

# Selection menu
doey_choose() {
  # args: list of options
  if [ "$HAS_GUM" = true ]; then
    printf '%s\n' "$@" | gum choose
  else
    local i=1
    for opt in "$@"; do
      printf "  %d) %s\n" "$i" "$opt"
      i=$((i + 1))
    done
    printf "  > "
    read -r choice
    # Return selected option
    local idx=1
    for opt in "$@"; do
      [ "$idx" -eq "$choice" ] && echo "$opt" && return 0
      idx=$((idx + 1))
    done
    return 1
  fi
}

# Spinner for long operations
doey_spin() {
  local title="$1"; shift
  if [ "$HAS_GUM" = true ]; then
    gum spin --spinner dot --title "$title" -- "$@"
  else
    printf "  %s..." "$title"
    "$@" >/dev/null 2>&1
    printf " done\n"
  fi
}

# Styled header
doey_header() {
  if [ "$HAS_GUM" = true ]; then
    gum style --foreground 212 --bold "$1"
  else
    printf "\033[1;36m%s\033[0m\n" "$1"
  fi
}
```

### Making Fallback Seamless

The key is that the plain-bash fallback must not feel "broken" — it should feel intentional:

1. **Same information, different rendering** — The fallback shows identical content, just without animations/colors/borders
2. **No "gum not found" warnings** — Silent detection, not nagging
3. **Existing ANSI styling preserved** — Doey already has `BRAND`, `SUCCESS`, `DIM`, etc. color codes. The fallback uses these
4. **Consistent UX contract** — Both paths return the same values, accept the same arguments

### Where gum adds most value vs. where fallback is fine

| Feature | gum Value | Fallback Quality |
|---------|-----------|-----------------|
| `gum choose` (session picker) | High — fuzzy filter, scroll | Acceptable — numbered list |
| `gum spin` (install progress) | High — animated spinner | Acceptable — static "..." |
| `gum confirm` (destructive ops) | Medium — styled Y/N | Good — `read -rp` is fine |
| `gum input` (text entry) | Medium — placeholder, validation | Good — `read -r` is fine |
| `gum style` (headers/banners) | Medium — borders, padding | Good — ANSI codes work |
| `gum format` (markdown) | Low — rarely needed in CLI | N/A |
| `gum table` (task display) | High — column alignment | Medium — printf formatting |

## 5. Recommended Install Strategy

### Decision: Optional dependency with auto-install offer

**Tier:** Same as jq — optional, with offer to install, silent fallback if declined.

### Implementation Plan

**In install.sh** (new step, between jq check and doey-tui build):

```bash
if has gum; then
  GUM_VER=$(gum --version 2>/dev/null || echo "unknown")
  check_ok "gum ${DIM}(v$GUM_VER)${RESET}"
else
  if ask_install "gum (Charmbracelet)"; then
    _install_gum  # Platform-aware installer (see below)
  else
    warn_msg "gum not found (optional — CLI will use plain text fallback)"
  fi
fi
```

**The `_install_gum()` function:**

```bash
_install_gum() {
  local GUM_VERSION="0.17.0"
  local os arch archive
  os=$(uname -s)
  arch=$(uname -m)

  case "$os/$arch" in
    Darwin/arm64)   archive="gum_${GUM_VERSION}_Darwin_arm64.tar.gz" ;;
    Darwin/x86_64)  archive="gum_${GUM_VERSION}_Darwin_x86_64.tar.gz" ;;
    Linux/x86_64)   archive="gum_${GUM_VERSION}_Linux_x86_64.tar.gz" ;;
    Linux/aarch64)  archive="gum_${GUM_VERSION}_Linux_arm64.tar.gz" ;;
    *)
      warn_msg "No pre-built gum binary for $os/$arch"
      detail "Install manually: https://github.com/charmbracelet/gum#installation"
      return 1
      ;;
  esac

  local url="https://github.com/charmbracelet/gum/releases/download/v${GUM_VERSION}/${archive}"
  local tmp_dir
  tmp_dir=$(mktemp -d)

  if curl -fsSL "$url" | tar xz -C "$tmp_dir" gum 2>/dev/null; then
    mv "$tmp_dir/gum" "$HOME/.local/bin/gum"
    chmod +x "$HOME/.local/bin/gum"
    rm -rf "$tmp_dir"
    printf "  ${SUCCESS}✓${RESET} gum installed to ~/.local/bin/gum\n"
  else
    rm -rf "$tmp_dir"
    # Fallback to brew if available
    if [ "$HAS_BREW" = true ]; then
      run_install "gum" "brew install gum"
    else
      warn_msg "Failed to download gum binary"
      detail "Install manually: brew install gum (macOS) or see https://github.com/charmbracelet/gum"
      return 1
    fi
  fi
}
```

### Version Pinning

- **Pin to a specific version** (e.g., `0.17.0`) in install.sh
- Update the version when we test a new release
- Don't auto-update — gum is a CLI tool, not a security-sensitive dependency
- The pinned version should be a constant at the top of install.sh for easy updates

### doey doctor Integration

Add to the doctor command:

```bash
if command -v gum >/dev/null 2>&1; then
  _doc_check ok "gum" "$(gum --version 2>/dev/null || echo 'installed')"
else
  _doc_check skip "gum not installed" "using plain text fallback"
fi
```

## 6. Summary / Recommendations

| Question | Answer |
|----------|--------|
| Should install.sh install gum? | Yes, with `ask_install` (user opt-in), not forced |
| Required or optional? | **Optional** — same tier as jq |
| Install method? | Direct binary download (4.5 MB), brew fallback |
| Version pinning? | Yes — pin to specific release, update manually |
| What if gum missing? | Silent fallback to plain `read`/`printf`/ANSI |
| Cross-platform? | macOS (arm64, x86_64), Linux (x86_64, arm64) |
| Where to put binary? | `~/.local/bin/gum` (same as doey itself) |
| Download impact? | ~4.5 MB one-time, ~3s on broadband |
| Runtime impact? | Zero if not installed; subprocess per invocation if installed |

### Architecture for wrapper functions

Create `shell/doey-ui.sh` as a sourced library with:
- `HAS_GUM` detection at source time
- Wrapper functions: `doey_confirm`, `doey_input`, `doey_choose`, `doey_spin`, `doey_header`, `doey_table`
- Source it from `doey.sh` and `install.sh`
- Each wrapper: gum path + plain bash path, identical return values
- All wrappers must be bash 3.2 compatible

This approach follows Doey's established patterns (jq/python3, doey-tui/info-panel.sh) and satisfies the fresh-install constraint.
