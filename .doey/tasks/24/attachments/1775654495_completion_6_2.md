  5. Passing Multi-Line Text Safely on the CLI — Recommendation

  Go's flag package treats flag values as opaque strings, so any approach works at the parser level. The constraint is the shell invocation from agent prompts / bash hooks.

  Options surveyed:

  │                    Approach                    │                       Pro                       │                                 Con                                 │
  │ --origin-prompt "$MSG" (double-quoted var)     │ Simple, idiomatic                               │ Breaks on embedded ", $, backticks — user text is adversarial       │
  │ --origin-prompt "$(cat <<'EOF' … EOF)" heredoc │ Handles quotes, newlines, $ (with quoted 'EOF') │ Still goes through one layer of arg-expansion; clunky in one-liners │
  │ --origin-prompt @file (Go reads file)          │ Cleanest for arbitrary bytes, reusable          │ Requires caller to write a temp file; new semantics to implement    │
  │ --origin-prompt-base64 <b64>                   │ Zero quoting hazards, single-line friendly      │ Extra encode/decode step; opaque in logs                            │
  │ Read from stdin when flag absent               │ No flag needed                                  │ Conflicts with other stdin uses; surprising                         │

  Recommendation — support two modes, parsed by runTaskCreate:

  1. --origin-prompt <string> for the common case (short single-line prompts, agent wraps in a 'single-quoted' literal after escaping any ' as '\'').
  2. --origin-prompt-file <path> (or equivalently --origin-prompt @<path>) for multi-line / untrusted content. Boss writes the raw user message to a temp file (e.g. "$RUNTIME_DIR/tmp/origin_$$.txt") with a single cat > "$f" <<'DOEY_EOF' … DOEY_EOF heredoc, then passes the path. The heredoc quoting with a 'DOEY_EOF' sentinel disables expansion entirely, so $/`/" are all safe. doey-ctl reads the
  file and unlinks it (or leaves it for Boss to trash).

  Boss template pattern to add:
  ORIGIN_FILE=$(mktemp "${RUNTIME_DIR}/tmp/origin.XXXXXX")
  cat > "$ORIGIN_FILE" <<'DOEY_ORIGIN_EOF'
  <paste user's verbatim message here>
  DOEY_ORIGIN_EOF
  TASK_ID=$(doey task create --title "…" --origin-prompt-file "$ORIGIN_FILE" …)
  trash "$ORIGIN_FILE"

  The two-flag approach keeps simple cases one-liner-friendly while guaranteeing correctness for anything containing quotes, newlines, or shell metacharacters. A base64 variant is unnecessary complexity given the file path already exists (RUNTIME_DIR/tmp/).
