# Passive Taskmaster — event-driven, not polling

## Goal
Replace Taskmaster's active polling loop with a reactive, event-driven model. Dispatch → sleep → wake only when triggered. Zero token burn between events.

## Problem
Current agent definition says "permanent active loop — never idle" with:
- Full state scan every cycle (messages, statuses, results, crashes, stale alerts)
- 5-10s sleep between cycles via taskmaster-wait.sh
- Even idle cycles consume tokens reading unchanged files
- Over a session, this bleeds thousands of tokens doing nothing

## Design: Reactive Taskmaster

### New model
1. **Receive task from Boss** → plan dispatch → send to team(s) → **sleep**
2. **Sleep = block on taskmaster-wait.sh** until a trigger fires
3. **Wake ONLY on:** new .msg file, FINISHED status, crash alert, stale alert, new queued task
4. **On wake:** read ONLY the triggering event(s), act on them, sleep again
5. **No scanning unchanged state.** If nothing triggered, nothing happens.

### What triggers wake (taskmaster-wait.sh already supports these):
- `$RUNTIME_DIR/triggers/<TASKMASTER_SAFE>.trigger` — Boss sent a message
- `FINISHED` status detected — a worker/team completed
- `crash_pane_*` file appeared — something crashed
- `stale_*` file appeared — heartbeat expired
- `QUEUED_TASKS` — new task arrived with no team assigned

### What changes:
1. **Agent definition** — remove "permanent active loop", replace with "reactive sleep-wake" model
2. **Remove the full-scan-every-cycle pattern** — replace with targeted reads based on wake reason
3. **taskmaster-wait.sh** — already has trigger support, may need to pass wake reason back
4. **Delta tracking** — remove sm_seen_results tracking (unnecessary if we only read on trigger)
5. **Dispatch flow** — stays the same (dispatch → set task status → sleep)
6. **Boss comms** — stays the same (.msg files + triggers)

## Steps
- [ ] Step 1: Rewrite Taskmaster agent definition — reactive model, no polling loop (high)
- [ ] Step 2: Enhance taskmaster-wait.sh — output wake reason so Taskmaster knows what triggered it (medium)
- [ ] Step 3: Remove delta-tracking boilerplate (sm_seen_results, "skip if unchanged") (low)
- [ ] Step 4: Add explicit "go to sleep" instruction after every action block (low)
- [ ] Step 5: Test — dispatch a task, verify Taskmaster sleeps after dispatch, wakes on completion (medium)

## Risks
- Taskmaster might miss events if wake reasons are incomplete: mitigate by keeping file-based triggers as fallback
- Crash recovery depends on periodic scanning: stale alerts from taskmaster-wait.sh cover this

## Scope
- **Files**: agents/doey-taskmaster.md.tmpl (or ~/.claude/agents/doey-taskmaster.md), .claude/hooks/taskmaster-wait.sh
- **Teams**: 1 team, 1-2 workers
- **Workers**: 2

## Acceptance Criteria
- Taskmaster agent definition has NO polling loop
- Taskmaster sleeps after every action (dispatch, report, commit)
- taskmaster-wait.sh outputs wake reason
- Token usage drops significantly between events
- Tasks still complete end-to-end (dispatch → work → commit → report)
