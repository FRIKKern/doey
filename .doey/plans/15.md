# Migrate doey.sh to Go — Phased Shell-to-Go Migration

## Goal
Replace the 3000-line doey.sh bash orchestrator with Go, making `doey` a single Go binary. Phased migration — one function group at a time, shell fallback during transition. Eliminates bash 3.2 compat issues, improves testability and error handling.

## Steps

### Phase 1 — Research & Inventory (complexity: low)
- [ ] Inventory all functions in doey.sh — name, line count, purpose, dependencies
- [ ] Group functions into logical domains (session launch, team creation, pane management, grid layout, worker spawning, status, CLI routing, etc.)
- [ ] Map inter-function dependencies (call graph)
- [ ] Identify which functions already have Go equivalents (doey-ctl, TUI, doey-router)
- [ ] Propose migration order based on: independence (fewer deps first), value (most fragile first), size (smallest first)
- [ ] Deliverable: migration roadmap document with ordered phases

### Phase 2 — Foundation: Go CLI skeleton + tmux library (complexity: medium)
- [ ] Create Go tmux wrapper package (split-pane, send-keys, set-option, capture-pane, etc.)
- [ ] Unified CLI entry point — doey binary handles all subcommands (already partially done via task #157)
- [ ] Shell-to-Go bridge: `doey` Go binary can call remaining shell functions as fallback
- [ ] Tests for tmux wrapper

### Phase 3 — Migrate status & pane management (complexity: medium)
- [ ] Status file read/write (already partially in doey-ctl)
- [ ] Pane resolution (dynamic pane lookup by role)
- [ ] Health checks and heartbeat management
- [ ] Context % monitoring

### Phase 4 — Migrate team creation & worker spawning (complexity: high)
- [ ] Team window creation (add-window, add-team)
- [ ] Worker spawning (Claude process launch with correct agent/flags)
- [ ] Grid layout calculations
- [ ] Batch boot workers

### Phase 5 — Migrate session lifecycle (complexity: high)
- [ ] Session creation (launch_session_dynamic)
- [ ] Dashboard setup (window 0)
- [ ] Core Team setup (window 1)
- [ ] Session stop/cleanup

### Phase 6 — Remove shell fallback (complexity: low)
- [ ] Delete doey.sh (or archive)
- [ ] Update install.sh — single Go binary install
- [ ] Update tests — remove bash compat tests for migrated code
- [ ] Final E2E validation

## Risks
- **Regression during migration**: Each phase could break running sessions. Mitigation: shell fallback stays active until Go equivalent is proven. Feature flags to switch between shell/Go per function group.
- **tmux interaction complexity**: Shell is natural for tmux; Go requires exec.Command wrappers. Mitigation: dedicated tmux Go package with thorough tests.
- **Scope creep**: 3000 lines is a lot. Mitigation: strict phase boundaries, each phase is independently shippable.
- **Concurrent development**: Other tasks modify doey.sh while migration is in progress. Mitigation: migrate smallest/most-stable functions first.

## Scope
- **Files**: shell/doey.sh (primary), tui/ (Go binary), shell/*.sh (supporting scripts)
- **Teams**: 1-2 teams per phase
- **Workers**: 2-4 per phase, 6 phases total
- **Timeline**: This is a multi-session initiative, not a single dispatch

## Acceptance Criteria
- Phase 1: Migration roadmap document with function inventory and dependency graph
- Phase 2: `doey` Go binary can create tmux sessions and panes
- Phase 3: All status operations work via Go (no shell fallback)
- Phase 4: `doey add-window` and `doey add-team` are pure Go
- Phase 5: `doey` launch creates full session without doey.sh
- Phase 6: doey.sh deleted, `doey doctor` passes, fresh install works
