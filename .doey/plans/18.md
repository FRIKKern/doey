# Rich subtask detail in Go TUI

## Goal
Each subtask in the task detail view should show rich information: status badge, assignee, timestamps, worker output, and expandable detail. Currently subtasks render as minimal "id:title:status" lines.

## Steps
- [ ] Step 1: Research current subtask rendering — find where subtasks are parsed and displayed in tui/ (model, view, store). Map data flow from SQLite/.task files to Go structs to Bubble Tea view. (complexity: low)
- [ ] Step 2: Extend subtask data model — ensure the Go Subtask struct carries: status, assignee (worker pane), created_at, completed_at, result summary. Parse these from TASK_SUBTASK_N_* fields during .task file sync and/or from the subtasks DB column. (complexity: medium)
- [ ] Step 3: Rich subtask list rendering — replace plain text with styled rows: status badge (colored checkmark/diamond/circle), assignee pane tag, title, and duration (completed_at - created_at). Use lipgloss for badges. (complexity: medium)
- [ ] Step 4: Worker output summary per subtask — link TASK_REPORT_N entries to subtasks (match by author pane or subtask ID). Show a one-line summary under each subtask row. (complexity: medium)
- [ ] Step 5: Expandable subtask detail — on Enter/click, expand inline or navigate to a detail view showing full context: attachments, decision log entries, reports, timestamps. (complexity: high)
- [ ] Step 6: Build verification + visual QA — build passes, subtasks render correctly with real task data. (complexity: low)

## Risks
- Data sparsity: Older tasks lack TASK_SUBTASK_N_* fields — must degrade gracefully to "id:title:status" format
- Report-to-subtask linking: No explicit foreign key between reports and subtasks — may need heuristic matching (author pane = assigned pane)
- Expandable UX complexity: Inline expansion in a list requires careful viewport/scroll management (past bugs in tasks 39, 173, 177)

## Scope
- Files: tui/internal/model/ (task detail, subtask structs), tui/internal/store/ (subtask queries), tui/internal/ui/ or view/ (rendering), possibly shell task sync
- Teams: 1 team, 2 workers max (focused TUI work)
- Workers: 2 — one for data model + store, one for view rendering

## Acceptance Criteria
- Subtask rows show status badge (colored), assignee pane, title, and duration
- Timestamps visible: created_at, completed_at, computed duration
- Worker output summary (one line) shown per subtask when available
- Enter/click on subtask expands to show full detail (attachments, logs, reports)
- Graceful degradation for tasks with minimal subtask data
- Go build passes clean
