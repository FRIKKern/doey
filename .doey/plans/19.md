# Fix Boss-to-Taskmaster message delivery end-to-end

## Goal
Messages sent via doey msg send never arrive at the recipient because pane IDs are stored/queried inconsistently. Fix the normalization so the SQLite message queue actually works. Keep send-keys as a nudge trigger (not the primary delivery). Add a 10-second post-send monitor that auto-nudges if the message isn't acknowledged.

## Steps
- [ ] Step 1: Research — trace the full message lifecycle. Where does doey msg send write pane IDs? Where does doey msg read query them? What format does each use? Find the exact mismatch. (complexity: low)
- [ ] Step 2: Normalize pane IDs — pick ONE canonical format (e.g., "1.0" short form). Ensure doey msg send normalizes --to and --from before DB insert. Ensure doey msg read normalizes --pane before DB query. Add a shared normalization function. (complexity: medium)
- [ ] Step 3: Fix existing messages — migration or cleanup query to normalize pane IDs already in the DB. (complexity: low)
- [ ] Step 4: Post-send monitor — after every doey msg send, wait 10 seconds, check if recipient status changed to BUSY. If still idle/sleeping, auto-nudge via send-keys (Escape + wake prompt). This should be a shell function or wrapper. (complexity: medium)
- [ ] Step 5: Update callers — Boss agent instructions, Taskmaster wait loop, and any skills that use doey msg send/read. Ensure they use the fixed API. (complexity: medium)
- [ ] Step 6: Verification — send a message from Boss to Taskmaster via doey msg send, confirm it appears in doey msg read on the Taskmaster side, confirm nudge fires if Taskmaster is sleeping. (complexity: low)

## Risks
- Changing pane ID format could break existing msg read calls in hooks/agents — must audit all callers
- The 10s monitor blocks the caller — should be backgrounded so Boss doesn't freeze
- Taskmaster wait loop triggers (taskmaster-wait.sh) must recognize the new message format

## Scope
- Files: doey-ctl Go source (msg send/read handlers), shell/doey.sh (if msg wrappers exist), hooks/taskmaster-wait.sh, agent templates that reference doey msg
- Teams: 1 team, 2 workers
- Workers: 2 — one for Go msg normalization, one for monitor + caller updates

## Acceptance Criteria
- doey msg send --to 1.0 messages are found by doey msg read --pane 1.0 (zero mismatches)
- Pane ID normalization is a single shared function (not duplicated)
- Post-send monitor auto-nudges after 10s if recipient hasn't acknowledged
- Monitor runs in background (non-blocking)
- Taskmaster wait loop wakes on new messages
- All existing callers updated
- No regressions in message delivery
