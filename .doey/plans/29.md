# Criteria of Success + Proof of Success

## Philosophy
"Proof of Completion" answers "did the code ship?" — wrong question.
**"Proof of Success"** answers "is the real problem solved?"

Every task gets:
- **Criteria of Success** — defined at creation. What "solved" looks like in concrete, testable terms
- **Proof of Success** — the AI's best effort to prove the criteria are met. Real output, real tests, real evidence
- **Human Verification Guide** — clear steps for the human to confirm. The human is the ultimate decision maker
- **Accept / Review Later** — human presses Accept → ready to deploy. Review Later → stays in queue

## Task Lifecycle (revised)

```
created → in_progress → proof_gathered → awaiting_human_review → [accepted → deploy] | [review_later]
```

The AI goes as far as it can to prove success:
- Run the build, capture output
- Run tests, capture output
- Try to reproduce the original bug and show it's gone
- Check the acceptance criteria one by one
- Generate a human verification guide for anything it can't prove itself

## Steps

- [ ] **Step 1: Task schema — add success_criteria and proof_of_success fields** (complexity: medium)
  - `.task` files: add `TASK_SUCCESS_CRITERIA=` field (set at creation)
  - Result JSON: replace `proof_type`/`proof_content` with structured `proof_of_success` object:
    ```json
    {
      "criteria_results": [
        {"criterion": "go build passes", "status": "pass", "evidence": "exit 0, no errors"},
        {"criterion": "bug no longer reproduces", "status": "pass", "evidence": "tested X → got Y"},
        {"criterion": "UI shows correct data", "status": "needs_human", "guide": "Open Tasks tab → verify all 324 tasks visible"}
      ],
      "human_verification_guide": "Steps the human should take to confirm...",
      "auto_verified_count": 2,
      "needs_human_count": 1
    }
    ```
  - SQLite schema: update proof columns to match
  - `doey task create` and `doey task update`: support success_criteria field

- [ ] **Step 2: Auto-verification in stop-results.sh** (complexity: medium)
  - When worker finishes, hook reads `TASK_SUCCESS_CRITERIA` from the task file
  - Runs what it can automatically: build, vet, test, grep for expected patterns
  - Captures structured results per criterion (pass/fail/needs_human + evidence)
  - Generates human verification guide for criteria it can't auto-verify
  - Writes structured proof_of_success to result JSON

- [ ] **Step 3: Worker agent instructions — verify success, not just completion** (complexity: low)
  - Workers must read the success criteria before starting
  - Before finishing: check each criterion, emit structured PROOF lines
  - If a criterion can't be auto-verified, write a clear human guide
  - Move instructions to top of agent, make mandatory

- [ ] **Step 4: TUI — display Criteria + Proof + Human Guide** (complexity: medium)
  - Expanded card shows success criteria as a checklist:
    - ✓ criterion (auto-verified) — with evidence expandable
    - ◯ criterion (needs human review) — with verification guide
    - ✗ criterion (failed) — with failure details
  - Accept / Review Later buttons (already partially exists as task 316)
  - Accept → status transitions to accepted → deployment pipeline
  - Review Later → stays in pending_user_confirmation

- [ ] **Step 5: Boss task creation — always set success_criteria** (complexity: low)
  - When Boss creates tasks, always define concrete success criteria
  - Criteria should be testable: "X works", "Y no longer happens", "Z output matches"
  - Vague criteria like "code is clean" are rejected — must be verifiable

## Acceptance Criteria
- Every new task has explicit success_criteria defined at creation
- Workers verify each criterion and capture real evidence
- TUI shows criteria checklist with pass/fail/needs_human status
- Human verification guide is clear and actionable
- Accept button marks task ready for deployment
