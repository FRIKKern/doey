# Fix startup flow — loading screen + Taskmaster unresponsive

## Goal
Two linked startup issues:
1. The Go loading screen (doey-loading) hangs for up to 120s because it waits for Boss + Taskmaster + 1 worker to reach BUSY/READY, but briefings are delayed until after all teams spawn
2. Taskmaster starts and receives its briefing but becomes unresponsive to subsequent tasks/messages

## Steps

### Part A: Loading screen exits too slowly
- [ ] Fix allKeyPanesReady() in tui/cmd/doey-loading/main.go — remove the worker requirement. Only require Boss + Taskmaster to be "ready". Workers come online asynchronously (complexity: low)
- [ ] Alternative/additional: have on-session-start.sh write READY instead of BOOTING for the initial status (complexity: low)
- [ ] Reduce the timeout from 120s to 30-45s (complexity: low)
- [ ] Rebuild the doey-loading binary after changes (complexity: low)

### Part B: Taskmaster unresponsive after boot
- [ ] Research: capture what Taskmaster does after receiving its briefing — does it enter the wait loop correctly? Does taskmaster-wait.sh detect new messages/triggers? (complexity: medium)
- [ ] Check if taskmaster-wait.sh trigger file detection works on fresh startup — is the trigger path correct? (complexity: medium)
- [ ] Verify the Taskmaster agent prompt handles the initial briefing and enters the reactive wait loop (complexity: medium)
- [ ] Fix whatever causes unresponsiveness (complexity: depends on findings)

## Risks
- Changing on-session-start.sh READY status could affect other components that rely on BOOTING to BUSY transition
- Taskmaster wait loop changes could cause busy-looping (regression from task 131)

## Scope
- Files: tui/cmd/doey-loading/main.go, .claude/hooks/on-session-start.sh, .claude/hooks/taskmaster-wait.sh, possibly agents/doey-taskmaster.md
- Teams: 1 team, research + implementation
- Workers: 2 workers (one per part, parallel)

## Acceptance Criteria
- Loading screen exits within 15-20s on a normal startup (not 120s timeout)
- Taskmaster responds to dispatched tasks immediately after startup
- No busy-loop regressions
- doey-loading binary rebuilt and installed
