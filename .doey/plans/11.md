# Bulletproof Hook Permissions + Agent Self-Awareness

## Goal
Fix the recurring hook/permission problems systemically. Agents keep getting blocked by hooks because: (1) hooks fail to identify roles correctly, (2) agent prompts don't document tool restrictions, (3) agents waste attempts on blocked tools. Make hooks robust AND agents self-aware.

## Steps
- [ ] Step 1: Audit all hooks for role detection robustness (complexity: medium)
  - Review on-pre-tool-use.sh, on-prompt-submit.sh, stop-status.sh, stop-notify.sh
  - Verify doey-roles.sh is sourced with fallback in EVERY hook that checks roles
  - Add defensive guards: if role vars are empty, log warning and allow (fail-open for workers, fail-closed for dangerous ops)
  
- [ ] Step 2: Audit all agent .md.tmpl files for tool restriction documentation (complexity: medium)
  - Check each agent template: doey-boss, doey-taskmaster, doey-subtaskmaster, doey-worker, doey-task-reviewer, doey-deployment, doey-doey-expert, doey-critic
  - Verify each explicitly lists what tools are BLOCKED for their role
  - Compare agent docs vs actual hook enforcement — find mismatches

- [ ] Step 3: Add explicit "Tool Restrictions" section to every agent template (complexity: medium)
  - Each .md.tmpl gets a clear ## Tool Restrictions section listing blocked tools
  - Must match the enforcement in on-pre-tool-use.sh exactly
  - Include WHY each restriction exists (so agents can work around it properly)
  - Include the CORRECT alternative (e.g. "Cannot Read source -> dispatch to worker")

- [ ] Step 4: Harden on-pre-tool-use.sh role detection (complexity: low)
  - Ensure doey-roles.sh sourcing has 3+ fallback paths (already done in task #134, verify it stuck)
  - Add explicit logging when role detection fails
  - Never default to Boss role on failure — default to Worker (least restrictive)

- [ ] Step 5: Regenerate templates + verify consistency (complexity: low)
  - Run expand-templates.sh
  - Diff generated .md vs .md.tmpl to verify
  - Run bash 3.2 compat tests

## Risks
- Template changes could break agent behavior if restrictions are documented incorrectly: mitigation = cross-reference with hook source code
- Changing default role on detection failure could loosen security: mitigation = only change default for non-dangerous operations

## Scope
- **Files**: agents/*.md.tmpl (8-13 files), .claude/hooks/on-pre-tool-use.sh, .claude/hooks/on-prompt-submit.sh, .claude/hooks/common.sh
- **Teams**: 1 team, 2-3 workers (research + template edits + hook hardening)
- **Workers**: 2-3

## Acceptance Criteria
- Every agent .md.tmpl has a ## Tool Restrictions section matching hook enforcement
- on-pre-tool-use.sh sources doey-roles.sh with fallback in all paths
- Role detection failure defaults to Worker, not Boss
- expand-templates.sh produces clean output
- Bash 3.2 compat passes
- No agent should ever be surprised by a hook block
