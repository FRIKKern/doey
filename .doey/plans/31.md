# Fix SQLite SQLITE_BUSY database locking errors

## Goal
Eliminate SQLITE_BUSY "database is locked" errors that occur when multiple Doey panes (stop hooks, status updates, task operations) write to SQLite concurrently via doey-ctl.

## Steps
- [ ] Step 1: Research current DB open/init code in store.go — check for WAL mode, busy_timeout, connection pooling settings (complexity: low)
- [ ] Step 2: Enable WAL journal mode via PRAGMA journal_mode=WAL at DB open time if not already set (complexity: low)
- [ ] Step 3: Set PRAGMA busy_timeout=5000 (5 seconds) at DB open time so writers wait instead of failing immediately (complexity: low)
- [ ] Step 4: Add retry-with-exponential-backoff wrapper for write operations if not already present — catch SQLITE_BUSY and retry up to 3 times (complexity: medium)
- [ ] Step 5: Ensure the SQLite driver DSN includes _journal_mode=WAL&_busy_timeout=5000 or equivalent pragma execution (complexity: low)
- [ ] Step 6: Test under concurrent load — run multiple doey-ctl commands in parallel and verify zero SQLITE_BUSY errors (complexity: medium)

## Risks
- WAL mode creates -wal and -shm files alongside the DB — ensure cleanup paths account for these
- busy_timeout of 5s could cause slow stop hooks if DB is heavily contended — 5s is a safe upper bound, actual waits will be milliseconds
- Changing journal mode on existing DB is safe but requires no active transactions at the time

## Scope
- **Files**: tui/internal/store/store.go (DB open/init), tui/internal/store/schema.go (schema init), possibly tui/cmd/doey-ctl/commands.go
- **Teams**: 1 worker team
- **Workers**: 1 (single focused change)

## Acceptance Criteria
- WAL journal mode enabled on DB open
- busy_timeout pragma set to >= 5000ms
- Retry logic present for write operations
- Running 10+ concurrent doey-ctl writes produces zero SQLITE_BUSY errors
- Existing tests pass (go build, go test)
