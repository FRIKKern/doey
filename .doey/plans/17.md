# Subtask Accountability Pipeline

## Goal
Workers must always be assigned to a specific subtask (not just a parent task). When a worker finishes, the subtask is checked off. Subtaskmaster reviews output before marking done. End-to-end: Subtaskmaster breaks task into subtasks, assigns each to a worker, worker reports completion, Subtaskmaster reviews and checks off.

## Steps
- [ ] Step 1: Update Subtaskmaster agent template — mandate subtask creation before dispatch, assign subtask ID to each worker, review worker output before marking subtask done (complexity: medium)
- [ ] Step 2: Update Worker agent template — worker must know its assigned subtask ID, report against that subtask on completion (complexity: low)
- [ ] Step 3: Update stop hooks (stop-results.sh / stop-notify.sh) — include subtask ID in completion payload so Subtaskmaster can match result to subtask (complexity: medium)
- [ ] Step 4: Update dispatch skill (doey-dispatch) — pass subtask ID when dispatching to a worker pane (complexity: low)
- [ ] Step 5: Verify doey-ctl subtask commands work for this flow — subtask add, subtask update, subtask list must support status transitions (complexity: low)
- [ ] Step 6: End-to-end test — dispatch a real task, verify subtask assignment, completion, review, and check-off (complexity: low)

## Risks
- Over-engineering: adding too much ceremony slows workers down. Mitigation: keep it lightweight — subtask ID passed as env var, not complex protocol
- Stop hook changes could break existing flow. Mitigation: additive changes only, test existing tasks still complete normally
- Task #116 was cancelled for similar scope. Mitigation: keep scope tight — agent template instructions + one env var, no new hooks or systems

## Scope
- **Files**: Subtaskmaster agent template, Worker agent template, stop-results.sh, stop-notify.sh, doey-dispatch skill, doey-ctl subtask commands
- **Teams**: 1 team, 2 workers max (agent templates + hooks are cohesive)
- **Workers**: 2

## Acceptance Criteria
- Subtaskmaster always creates subtasks before dispatching workers
- Each worker knows its subtask ID and reports against it
- Subtaskmaster reviews worker output before checking off subtask
- Parent task progress reflects subtask completion
- Existing task flow (without subtasks) still works
