# Ephemeral Task-Sized Teams

## Goal
Teams spawn on demand, sized to the task's subtask count, and despawn when the task completes. Only Core Team (window 1) and Reserved Freelancers persist. No pre-spawned idle teams.

## Current State
- `add_dynamic_team_window()` in doey.sh spawns teams with fixed worker counts (DOEY_INITIAL_WORKER_COLS * 2)
- `kill_team_window()` exists but is only called manually via `doey kill-window`
- Taskmaster agent prompt already has auto-spawn logic (capacity check → `/doey-add-window`)
- `stop-notify.sh` sends `task_complete` to Taskmaster when Subtaskmaster finishes
- Startup pre-spawns teams based on wizard or DOEY_INITIAL_TEAMS config
- No auto-despawn mechanism exists

## Steps
- [ ] Step 1: Add `--workers N` flag to `doey add-window` CLI so Taskmaster can request exact worker count (complexity: low)
  - Currently only `--grid NxM` which forces 2-row layout. Need `--workers 3` to spawn exactly 3 workers
- [ ] Step 2: Add `--name "Task 225"` flag to `doey add-window` to name the team window after the task (complexity: low)
  - `tmux rename-window` already works, just needs CLI plumbing
- [ ] Step 3: Update Taskmaster agent prompt — on new task, count subtasks, spawn right-sized team, dispatch (complexity: medium)
  - Replace "find idle team" with "spawn team for this task"
  - Team size = max(subtask_count, 1), capped at DOEY_MAX_WORKERS
  - Pass task ID so team can be tracked
- [ ] Step 4: Update Taskmaster agent prompt — on task_complete from Subtaskmaster, despawn the team (complexity: medium)
  - When a team's task is done/confirmed: run `/doey-kill-window <window>`
  - Skip if team is Core Team (window 1) or Reserved Freelancer
  - Log despawn to activity
- [ ] Step 5: Update startup flow — start with Core Team only, no worker teams pre-spawned (complexity: medium)
  - Set DOEY_INITIAL_TEAMS=0 as default (was 1)
  - Wizard can still override for manual setup
  - First task triggers first team spawn
- [ ] Step 6: Update Boss agent prompt — include subtask count estimate in task dispatch messages (complexity: low)
  - Help Taskmaster right-size teams by including "WORKERS_NEEDED: N" in dispatch
- [ ] Step 7: Track team↔task mapping in session.env or team_N.env (complexity: low)
  - Add TASK_ID field to team env so despawn knows when task is done
  - Taskmaster can query which team owns which task

## Risks
- Race condition: task completes while workers still writing files. Mitigation: despawn waits for all workers FINISHED status before killing window
- Startup feels empty with no teams. Mitigation: loading screen shows "Awaiting first task..." until Boss dispatches
- Frequent spawn/despawn may be slow. Mitigation: team spawn is already ~5s, acceptable for task boundaries
- Existing sessions have persistent teams. Mitigation: behavior change only applies to NEW tasks; existing teams are left alone

## Scope
- **Files**: agents/doey-taskmaster.md (dispatch + despawn logic), agents/doey-boss.md (subtask count in dispatch), shell/doey.sh (add-window --workers/--name flags, startup defaults), team env files
- **Teams**: 2 workers (agent prompts + shell changes)
- **Workers**: 2

## Acceptance Criteria
- New task → Taskmaster spawns a team sized to subtask count
- Task complete → team despawns automatically (unless Core/Freelancer)
- Startup shows Core Team only, no idle worker teams
- `doey add-window --workers 3 --name "Task 225"` works from CLI
- Team env tracks which task it serves
