# doey-ctl flag API normalization

## Goal
Normalize all doey-ctl subcommand flags so callers use one consistent interface.

## Findings (from verification)
- Three different flag names for project directory: `--runtime`, `-dir`, `-project-dir`
- `db-status set` requires 4 fields — callers often only have pane + status
- `db-msg list` requires `--to` filter, no way to list all messages
- `db-status list` requires `--window-id`, no way to list all statuses
- File-based commands use `--` double-dash, DB commands use `-` single-dash

## Steps
- [ ] Step 1: Normalize all project-dir flags to `-project-dir` across db-* commands (low)
- [ ] Step 2: Make `db-status set` accept just `-pane-id` + `-status` (derive window/role from pane ID) (medium)
- [ ] Step 3: Make list commands work without required filters (show all, filter optionally) (low)
- [ ] Step 4: Update shell helpers that call doey-ctl to use normalized flags (low)
- [ ] Step 5: go build + verify (low)

## Risks
- Breaking existing callers: keep old flags as aliases during transition

## Scope
- **Files**: tui/cmd/doey-ctl/*.go (store_*_cmds.go files)
- **Teams**: 1 team, 2 workers max
- **Workers**: 2

## Acceptance Criteria
- All db-* commands accept `-project-dir` consistently
- `db-status set` works with just pane-id + status
- List commands work without required filters
- go build passes
- Shell helpers work with normalized flags
