# Doey Product Roadmap

## 1. Intent & Vision

Doey is becoming the **task-obsessed multi-agent coordination layer** for Claude Code. Every action traces back to a task. Every worker knows what it's doing and why. The system scales from a single developer to orchestrated teams — all through tmux, bash, and Claude Code's native primitives.

### Core Principles

- **Tasks are the atom of work** — No work happens without a task. Tasks have lifecycle, ownership, dependencies, and results
- **Coordination over raw parallelism** — 3 focused workers beat 10 unfocused ones. The Manager is the quality gate
- **Zero-config start, infinite ceiling** — `doey` in any directory gets you going. Teams, worktrees, research workflows layer on top
- **Ship in the repo** — Everything that matters lives in version control. Local state is ephemeral and reproducible

### What Success Looks Like

```
$ doey
→ Dashboard shows 3 active tasks, 2 workers busy, 1 completing
→ Boss receives "add OAuth support" → creates task → SM routes to team
→ Manager breaks into subtasks → dispatches to workers → validates results
→ Task board updates in real-time in the TUI
→ User sees progress without ever leaving the terminal
```

## 3. Mid-Term (1–3 Months)

### Smarter Worker Coordination

- [ ] **Dependency-aware dispatch** — don't send task B until task A completes
- [ ] **Worker affinity** — prefer sending file-related tasks to workers who recently read that file
- [ ] **Load balancing** — track worker context window usage, prefer workers with more headroom
- [ ] **Automatic scaling** — spawn additional workers when queue depth exceeds threshold
- [ ] **Worker health monitoring** — detect stuck/crashed workers, auto-restart with context

### Context Management

- [ ] **Context budget tracking** — each worker reports estimated tokens used
- [ ] **Smart compaction** — preserve task-relevant context during `/compact`
- [ ] **Cross-worker knowledge sharing** — workers can query a shared knowledge base
- [ ] **File ownership registry** — track which worker "owns" which files to prevent conflicts

```bash
# Example: context budget check before dispatch
if [ "$worker_context_pct" -gt 80 ]; then
    send_keys "$pane" "/compact"
    wait_for_ready "$pane"
fi
dispatch_task "$pane" "$task_id"
```

### Research Workflows

- [ ] **Multi-source research** — dispatch research to multiple workers, merge findings
- [ ] **Research reports** — structured output format for research tasks
- [ ] **Citation tracking** — research results link back to source files/URLs
- [ ] **Research cache** — don't re-research topics already investigated this session

### Improved Communication

- [ ] **Structured IPC messages** — JSON-based inter-pane communication replacing raw send-keys
- [ ] **Message queuing** — workers can receive messages while busy, process when ready
- [ ] **Broadcast channels** — topic-based pub/sub for cross-team coordination
- [ ] **Status streaming** — real-time task progress updates (not just BUSY/READY)

## 5. Architecture Decisions

### Decided

| Decision | Choice | Rationale | Date |
|----------|--------|-----------|------|
| IPC mechanism | File-based + send-keys | No dependencies, works everywhere, tmux-native | 2025-12 |
| Runtime state location | `/tmp/doey/<project>/` | Ephemeral by design, clears on reboot | 2025-12 |
| Config format | Bash source files | Zero-dep, native to the shell, easy to override | 2026-01 |
| Task storage | `.doey/tasks/` JSON files | Git-trackable, no database, simple tooling | 2026-02 |
| Plan storage | `.doey/plans/` Markdown | Human-readable, version-controllable, rich formatting | 2026-04 |
| Hook architecture | Bash scripts with exit codes | Simple contract (0/1/2), composable, debuggable | 2025-12 |
| Agent definitions | YAML frontmatter in Markdown | Claude Code native format, self-documenting | 2025-12 |
| Shell compatibility | Bash 3.2+ | macOS ships 3.2, must work without brew/upgrades | 2025-12 |

### Under Discussion

| Decision | Options | Considerations |
|----------|---------|----------------|
| TUI framework | Go (bubbletea) vs Rust (ratatui) vs Bash (tput) | Go chosen for bubbletea ecosystem, but Bash keeps zero-dep promise |
| Worker isolation | Worktrees vs branches vs copies | Worktrees are git-native but add complexity |
| Message format | JSON vs plain text vs protobuf | JSON balances structure with readability |
| Plugin loading | Dynamic source vs subprocess vs symlink | Must not break fresh-install guarantee |

## 6. Success Criteria

### Quantitative Goals

| Metric | Current | Target | Timeline |
|--------|---------|--------|----------|
| Fresh install time | ~30s | <15s | Near-term |
| Task dispatch latency | ~3s | <1s | Mid-term |
| Worker utilization | ~40% | >70% | Mid-term |
| Context window waste | ~30% | <15% | Mid-term |
| Concurrent workers supported | 6 | 12+ | Long-term |
| `doey doctor` pass rate (fresh) | 95% | 100% | Near-term |

### Qualitative Goals

- [ ] **Zero-friction onboarding** — new user productive in <5 minutes
- [ ] **Self-healing** — system recovers from worker crashes without user intervention
- [ ] **Transparent coordination** — user can always see what every worker is doing and why
- [ ] **Predictable behavior** — same input produces same task breakdown every time
- [ ] **Minimal footprint** — no background processes when doey isn't active

### Definition of Done (per phase)

**Near-term done when:**
- Plans render in TUI with scrolling
- Task subtrees work end-to-end
- `doey doctor` passes on fresh macOS and Linux installs

**Mid-term done when:**
- Dependency-aware dispatch handles 3+ task chains
- Context budget prevents worker OOM
- Research workflow produces structured reports

**Long-term done when:**
- Plugin installed from registry works after fresh install
- Multi-repo workspace coordinates 3+ repos
- Session survives terminal restart with task continuity
