# Worker Task Accountability — mandatory task assignment and completion reporting

## Goal

Every worker must be assigned to a task before starting work, and must report on that task (subtask status updated, result attached) before being allowed to go idle. No untracked work. No silent completions.

## Current State

- `on-prompt-submit.sh` extracts `Task #N` via regex from free-text prompts — fragile, fails if dispatch doesn't include that pattern
- `stop-status.sh` sets FINISHED and calls `task_update_status` to mark task "done" — but never checks if subtask was updated
- `stop-results.sh` attaches result JSON to task file — but only if task_id was extracted successfully
- Dispatch skill (`doey-dispatch/SKILL.md`) says "Include Task #N in the prompt header" — advisory, not enforced
- Subtaskmaster agent definitions don't mechanically guarantee task_id propagation
- No hook blocks a worker from starting without a task assignment

## Steps

### Phase 1: Structured task_id propagation (eliminate regex guessing)

- [ ] **Step 1.1**: Add `DOEY_TASK_ID` env var injection to dispatch flow (high)
  - When Subtaskmaster dispatches to a worker, write `task_id` to `${RUNTIME_DIR}/status/${PANE_SAFE}.task_id` BEFORE sending the prompt
  - Update `doey-dispatch` skill to require this step
  - Update `doey-delegate` skill similarly
  - This makes task_id available to all hooks without regex extraction

- [ ] **Step 1.2**: Harden `on-prompt-submit.sh` task_id extraction (medium)
  - Read from `${RUNTIME_DIR}/status/${PANE_SAFE}.task_id` as PRIMARY source (already written by dispatch)
  - Keep `Task #N` regex as FALLBACK (for manual send-keys)
  - Also support structured header format: `[TASK:N]` at start of prompt
  - Log warning if no task_id found for worker prompts

- [ ] **Step 1.3**: Block workers without task assignment (medium)
  - In `on-prompt-submit.sh`: if `is_worker` and no task_id found from any source, return exit code 2 with feedback: "No task assigned. Dispatch must include Task #N or pre-register task_id."
  - Exception: `/compact`, `/doey-*` slash commands, reserved panes
  - Exception: First prompt after launch (READY→BUSY, no prior task_id file)

### Phase 2: Mandatory completion reporting (enforce before FINISHED)

- [ ] **Step 2.1**: Add subtask auto-update in `stop-status.sh` (high)
  - When worker finishes with a task_id: check if any subtask in the `.task` file references this pane (by `TASK_SUBTASK_N_WORKER` field)
  - If found and still `pending`/`in_progress`: auto-update to `done` (or `failed` if error detected)
  - If no subtask linked to this pane: log warning but don't block (Subtaskmaster may track differently)

- [ ] **Step 2.2**: Ensure result JSON always links to task (medium)
  - `stop-results.sh` already does this when task_id is present — verify it works with the new propagation from Phase 1
  - Add fallback: if `.task` file not found for task_id, log error but don't crash

- [ ] **Step 2.3**: Add completion summary to task file (medium)
  - In `stop-results.sh`: call `task_add_report` with worker completion summary (status, files changed, tool count)
  - This creates a permanent audit trail in the `.task` file, not just an attachment

### Phase 3: Dispatch contract enforcement

- [ ] **Step 3.1**: Update `doey-dispatch` skill — mandatory pre-registration (medium)
  - Before `send-keys`, skill MUST write task_id file: `echo "$TASK_ID" > ${RUNTIME_DIR}/status/${PANE_SAFE}.task_id`
  - Before `send-keys`, skill MUST register subtask: `doey_task_add_subtask "$PD" "$TASK_ID" "$SUBTASK_TITLE" "$ASSIGNEE" "$PANE_ID"`
  - Prompt format must start with `[TASK:N]` structured header

- [ ] **Step 3.2**: Update `doey-delegate` skill similarly (low)

- [ ] **Step 3.3**: Update Subtaskmaster agent template — dispatch contract (medium)
  - Add explicit instruction: "Before dispatching to any worker, you MUST: (1) write task_id file, (2) register subtask, (3) include [TASK:N] header"
  - This is advisory (agent prompt) but combined with hook enforcement creates defense-in-depth

## Risks

- **Blocking legitimate work**: A worker dispatched manually (not through skill) might get blocked
  - Mitigation: Keep `Task #N` regex fallback, only block if NO task_id source found
- **Race condition**: Task_id file written by Subtaskmaster, read by worker's on-prompt-submit — timing
  - Mitigation: File is written BEFORE send-keys, hook reads AFTER prompt arrives — natural ordering
- **Subtask linking**: Not all tasks use `TASK_SUBTASK_N_WORKER` field consistently
  - Mitigation: Auto-update is best-effort, log warning, don't block on missing link
- **First-prompt exception**: Worker's first prompt after launch has no task context
  - Mitigation: Skip enforcement if no `.task_id` file exists AND status was READY/RESERVED

## Scope

- **Files**:
  - `.claude/hooks/on-prompt-submit.sh` — task_id validation
  - `.claude/hooks/stop-status.sh` — subtask auto-update
  - `.claude/hooks/stop-results.sh` — completion report
  - `.claude/skills/doey-dispatch/SKILL.md` (or `.md.tmpl`) — pre-registration
  - `.claude/skills/doey-delegate/SKILL.md` (or `.md.tmpl`) — pre-registration
  - `agents/doey-subtaskmaster.md.tmpl` — dispatch contract
- **Teams**: 1 team, 2-3 workers (hooks are non-overlapping files)
- **Workers**: 2-3

## Acceptance Criteria

- Worker cannot start work without a task_id (hook blocks with feedback)
- On worker completion, subtask is auto-updated to done/failed
- Result JSON is always linked to task file when task_id exists
- Completion report (task_add_report) written on every worker stop
- `Task #N` regex still works as fallback for manual dispatch
- Bash 3.2 compatible — `tests/test-bash-compat.sh` passes
- No regression: existing dispatch flow still works
