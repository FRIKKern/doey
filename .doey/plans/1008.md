# Masterplan: Elevate Doey's Planning Process

**Task:** #508  
**Status:** Complete

**Research Sources:** Cursor Plan (web), Oh My Codex (web + GitHub), /deep-interview (codebase), Doey masterplan infra (codebase)

## Executive Summary

Doey's masterplan system works but is primitive compared to Cursor Plan's polished UX and Oh My Codex's rigorous methodology. The key insight: **Cursor teaches us how plans should LOOK and FEEL, OmX teaches us how plans should be MADE, and our own /deep-interview shows we already have the foundation for structured requirements gathering.** The gap is stitching these together.

Three high-impact improvements:
1. **Real-time plan rendering** inspired by Cursor's streaming markdown with interactive checkboxes
2. **Multi-perspective planning** inspired by OmX's Planner/Architect/Critic consensus (RALPLAN-DR)
3. **Interview-first workflow** using our existing /deep-interview as the mandatory first phase

## Research Findings

### 1. Cursor Plan — Real-Time Rendering UX

**What Cursor Does:**
Cursor's Plan feature (introduced in v2.1, evolved through v2.2 and v3.0) is a "plan-before-code" approach where the AI generates a structured plan BEFORE writing any code.

**Key UX Patterns:**

| Pattern | How It Works | Why It's Effective |
|---------|--------------|-------------------|
| **Streaming Markdown** | Plan renders in real-time as the AI generates it. Users see sections appear progressively — headers, then bullet points, then details | Reduces perceived wait time. Users can start reading/evaluating before generation completes |
| **Interactive Checkboxes** | Each plan step becomes a checkbox. Users can check/uncheck, reorder, edit, or delete steps before execution | Gives users control over WHAT gets implemented. Reduces "AI did the wrong thing" |
| **Plan → Agent Pipeline** | After plan approval, user clicks "Send to Agent" to execute. Plan steps become the agent's task list | Clean handoff from planning to implementation. The plan IS the spec |
| **Inline Editing** | Users can modify the plan text directly — add constraints, change approach, refine scope | Collaborative feel. The plan is a living document, not a take-it-or-leave-it output |
| **Progressive Disclosure** | Starts with high-level steps, expands into detail on demand | Prevents information overload. Users see the forest before the trees |
| **Worktree Parallelism (v3.0)** | Multiple plan steps can execute in parallel via isolated git worktrees | Faster execution without merge conflicts |
| **Clarifying Questions** | Before planning, Cursor may ask 1-3 targeted questions about ambiguous requirements | Prevents wrong-direction planning. Quick, focused, not a full interview |

**Visual Design:**
- Plan appears in a dedicated sidebar panel (not inline in chat)
- Markdown-native rendering with syntax highlighting for code blocks
- Clear visual hierarchy: numbered steps → sub-steps → details
- Status indicators on each step (pending, running, complete, failed)
- Collapsible sections for large plans

**What Doey Can Learn:**
- Real-time rendering is the #1 UX differentiator — plan should stream visually as it's being written
- Checkboxes/interactive elements give users agency over the plan
- The plan file should be the SPEC that drives execution — not a separate artifact
- Clarifying questions before planning are brief (2-3), not a full interview

### 2. Oh My Codex (OmX) — Masterplan Methodology

*(Full research in /tmp/doey/doey/research-508/worker-3.md — 266 lines)*

**What OmX Does:**
OmX is an orchestration layer for Codex CLI (2,867 GitHub stars, by Yeachan Heo). Its planning system is its crown jewel.

**The Staged Pipeline:**
```
$deep-interview → $ralplan → $team/$ralph → verify → fix
```

**RALPLAN-DR Consensus — The Core Innovation:**

Three roles deliberate before any code is written:

| Role | Focus | Question |
|------|-------|----------|
| **Planner** | Task breakdown, sequencing, effort | What needs to happen, in what order? |
| **Architect** | Technical design, tradeoffs, structure | How should this be built? |
| **Critic** | Risk, feasibility, missed cases | What could go wrong? |

This produces plans that are **decomposed** (Planner), **technically sound** (Architect), and **stress-tested** (Critic).

**Key OmX Principles:**
1. **Clarify before coding** — Socratic interview to prevent wrong-direction work
2. **Consensus before execution** — Three perspectives must align
3. **Blocking artifacts** — PRD + Test Spec required before implementation is allowed
4. **Verification gates** — Explicit completion conditions between phases
5. **Persistent state** — Plans survive context resets
6. **Observable state** — Live planning state for dashboards
7. **Two intensity levels** — Quick consensus vs deliberate mode for high-risk changes

**What Doey Can Learn:**
- Multi-perspective deliberation produces MUCH better plans than single-agent planning
- Hard gates between phases (plan → PRD → exec) prevent premature implementation
- PRD + Test Spec as blocking artifacts is a powerful quality lever
- The Planner/Architect/Critic pattern maps naturally to Doey's multi-agent architecture

### 3. Doey's /deep-interview — Existing Foundation

**What We Already Have:**

The `/deep-interview` skill implements a structured interview protocol:

**Protocol:**
1. **Interviewer agent** (doey-interviewer) runs a structured conversation
2. **Researcher agent** (doey-interview-researcher) investigates codebase on demand
3. Interview follows phases: understand intent → identify ambiguities → clarify scope → establish constraints → define success criteria
4. Output: A structured **brief** with clear intent, scope, constraints, and deliverables
5. Brief is written to a file and monitored by a **watcher script** for downstream consumption

**Interview Team Setup:**
- Dedicated interview team with interviewer + researcher panes
- Researcher can explore codebase while interviewer maintains conversation flow
- Brief output is structured markdown with standard sections

**Strengths:**
- Already exists and works in Doey's multi-agent model
- Researcher sub-agent is a powerful pattern — can investigate while interviewing
- Structured brief output could feed directly into planning phase
- Watcher mechanism enables async handoff to downstream phases

**Gaps:**
- Currently standalone — not integrated into the masterplan workflow
- No automatic handoff from interview → planning
- Interview depth is fixed — no "quick mode" for simple requests
- Output format doesn't directly map to a plan structure

### 4. Doey's Current Masterplan Infrastructure

**How It Works Today:**

| Component | Implementation | Notes |
|-----------|---------------|-------|
| **Entry point** | `doey plan <goal>` → `_create_masterplan_team()` in doey.sh | Creates a dedicated team window |
| **Team layout** | Planner (pane 0) + Plan Viewer (pane 1) + Workers (panes 2-5) | 6-pane grid |
| **Plan viewer** | Shell script that watches the plan file and re-renders on change | Uses `inotifywait` or polling fallback |
| **Plan file** | Markdown at `.doey/plans/masterplan-<timestamp>.md` | Free-form structure |
| **Planner agent** | Has a system prompt (this agent) that defines the workflow | Research → Synthesize → Write |
| **Plan rendering** | glow or bat for terminal markdown rendering | Re-renders on file change |

**Current Workflow:**
1. User runs `doey plan "goal"`
2. Masterplan team is created with Planner + Viewer + 4 Workers
3. Planner asks clarifying questions (manual, ad hoc)
4. Planner dispatches research workers
5. Planner synthesizes and writes plan to `.doey/plans/`
6. Viewer pane renders plan in real-time as file changes
7. User reviews, asks for changes
8. When ready, tasks are created from plan sections

**Current Gaps:**
1. **No structured interview phase** — Planner asks ad hoc questions, not a protocol
2. **No multi-perspective planning** — Single planner, no architect/critic deliberation
3. **No quality gates** — Plan can go to execution without review
4. **No interactive plan elements** — Viewer is read-only, no checkboxes or editing
5. **Plan format is free-form** — No standard structure or required sections
6. **No blocking artifacts** — Can execute without PRD or test spec
7. **Research workers can't persist findings** — Write tool is blocked for workers writing to certain paths (observed during this session)
8. **No plan versioning** — Overwrite on update, no diff/history

## Improvement Plan

### Phase 1: Interview-First Workflow (Quick Win)

**Goal:** Integrate /deep-interview as the mandatory first phase of masterplan creation.

**Changes:**
1. When `doey plan <goal>` is run, launch /deep-interview BEFORE creating the research team
2. Interview brief output feeds automatically into the Planner's context
3. Add a "quick mode" for simple goals (2-3 questions max) vs "deep mode" for complex ones
4. Brief watcher triggers the transition from interview → research phase

**Effort:** Medium — mostly wiring existing components together  
**Impact:** High — prevents wrong-direction research and planning

### Phase 2: Real-Time Plan Rendering (UX Upgrade)

**Goal:** Make the plan viewer feel like Cursor's streaming plan experience.

**Changes:**
1. **Streaming render** — Plan viewer re-renders on every file write, not just on save. Use `fswatch`/`inotifywait` with debounce
2. **Structured plan format** — Define a standard plan schema:
   ```markdown
   # Plan: <title>
   ## Context (from interview brief)
   ## Approach
   ### Step 1: <name>
   - [ ] Sub-task A
   - [ ] Sub-task B
   ### Step 2: <name>
   ...
   ## Risks & Mitigations
   ## Success Criteria
   ## Test Strategy
   ```
3. **Status indicators** — Each step shows status (planned/in-progress/done/failed) via emoji or color
4. **Section-level updates** — Planner can update individual sections without rewriting the whole file
5. **glow/bat with live reload** — Ensure the viewer uses a tool that handles partial-file updates gracefully

**Effort:** Medium  
**Impact:** High — transforms plan viewer from "file watcher" to "live dashboard"

### Phase 3: Multi-Perspective Planning (RALPLAN-inspired)

**Goal:** Add Planner/Architect/Critic deliberation to produce higher-quality plans.

**Changes:**
1. **Three-role synthesis** — After research completes, three workers deliberate:
   - **Planner Worker**: Breaks goal into phases, sequences tasks, estimates effort
   - **Architect Worker**: Reviews technical approach, identifies tradeoffs, suggests alternatives
   - **Critic Worker**: Challenges assumptions, identifies risks, finds missed edge cases
2. **Consensus protocol** — All three perspectives are synthesized by the Subtaskmaster into a unified plan
3. **Two intensity levels:**
   - **Standard** (default): Quick single-pass deliberation
   - **Deliberate** (for high-risk): Multiple rounds with explicit conflict resolution
4. **Blocking artifacts** — Plan must include both "What to build" (PRD-equivalent) and "How to verify" (test strategy) before execution is allowed

**Effort:** High — new workflow, new agent prompts, consensus protocol  
**Impact:** Very High — fundamentally better plans through multi-perspective validation

### Phase 4: Interactive Plan Elements (Advanced)

**Goal:** Let users interact with the plan, not just read it.

**Changes:**
1. **TUI plan editor** — Go-based TUI (extends existing tui/ codebase) that renders the plan with:
   - Checkboxes for toggling steps on/off
   - Drag-to-reorder steps
   - Inline editing of step descriptions
   - Expand/collapse for detail levels
2. **Plan → Task pipeline** — "Execute" command that converts checked plan steps into Doey tasks
3. **Plan versioning** — Git-tracked plan files with diff view between versions
4. **Parallel execution hints** — Plan steps can be marked as parallelizable, feeding into worker dispatch

**Effort:** Very High — new TUI component, significant engineering  
**Impact:** High — but Phase 1-3 deliver most of the value

## Key Insights

### From Cursor Plan
> **Real-time rendering IS the product.** A plan that appears progressively feels collaborative. A plan that appears all-at-once feels like a report. The streaming experience is what makes users feel IN the planning process, not just reviewing its output.

### From Oh My Codex
> **Three perspectives beat one.** The Planner/Architect/Critic pattern is OmX's superpower. One agent plans, another validates the design, a third stress-tests it. This catches blind spots that a single planner always misses.

### From /deep-interview
> **We already have the hardest piece.** Structured interviewing with a researcher sub-agent is sophisticated — most tools don't have this. The gap is just wiring it into the masterplan flow.

### From Current Infrastructure
> **The foundation is solid.** File-watcher plan viewer, dedicated team window, worker dispatch — these all work. The improvements are about PROTOCOL (what happens) and UX (how it looks), not infrastructure.

## Success Criteria

1. `doey plan "goal"` starts with 2-3 clarifying questions (interview phase) before any research
2. Plan renders in real-time in the viewer pane as it's being written

4. Each plan step shows status (planned → in-progress → done)
5. Plans are demonstrably better quality than current ad-hoc approach (measured by: fewer wrong-direction tasks, better risk coverage, clearer success criteria)
