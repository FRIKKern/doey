# Boss Interaction Log — capture, store, and organize every user message

## Goal
Capture and persistently store every message sent to Boss with rich metadata: timestamp, associated task IDs, message type (command, question, feedback, status check), session context. This interaction history reveals HOW the user works with Doey — which patterns repeat, what gets asked manually that should be automatic, where friction lives. This is the foundation for making Doey proactively smart.

## Why This Matters
Right now user interactions are ephemeral — they vanish after the conversation. We lose:
- What the user asks most often (automation candidates)
- How long between task dispatch and user follow-up (satisfaction signal)
- Which tasks generate the most back-and-forth (complexity signal)
- What the user does manually that Doey should handle (feature discovery)

## Steps
- [ ] Step 1: Research current message infrastructure — how doey msg works, SQLite schema for messages, what's already captured vs lost (complexity: low)
- [ ] Step 2: Design the interaction_log schema — SQLite table with: id, timestamp, session_id, task_id (nullable), message_text, message_type (command/question/feedback/status/other), source (user/taskmaster/worker), context (what was happening when sent) (complexity: medium)
- [ ] Step 3: Implement capture hook — intercept every Boss-bound message at the hook level (on-prompt-submit or similar) and log to SQLite via doey-ctl (complexity: medium)
- [ ] Step 4: Add doey-ctl interaction subcommands — log, list, search, stats (complexity: medium)
- [ ] Step 5: Auto-classify message type — simple heuristic: starts with question mark = question, contains task ID = task-related, matches known command patterns = command, etc. (complexity: medium)
- [ ] Step 6: Auto-link to active tasks — when a message arrives, check active tasks and tag the interaction with relevant task IDs based on context (complexity: medium)
- [ ] Step 7: Add TUI Interactions tab or section — chronological view of all user interactions, filterable by task, type, time range (complexity: high)
- [ ] Step 8: Build basic analytics — most common message types, avg messages per task, busiest periods, repeated questions (complexity: medium)

## Risks
- Privacy: storing raw user messages needs care — keep it local-only, never transmit (mitigation: SQLite in project .doey/ dir, no network)
- Performance: logging on every prompt must be fast and non-blocking (mitigation: async write, doey-ctl is fast Go binary)
- Classification accuracy: auto-classifying message types will be imperfect initially (mitigation: start with simple heuristics, improve over time)
- Schema bloat: must not duplicate existing message DB data (mitigation: research current schema first, extend rather than duplicate)

## Scope
- **Files**: doey-ctl (Go — new interaction subcommand + SQLite table), on-prompt-submit hook, Boss agent prompt, TUI interaction view (Go)
- **Teams**: 1 team
- **Workers**: 2-3 (schema+capture worker, TUI worker, analytics worker)

## Acceptance Criteria
- Every user message to Boss is persisted in SQLite with timestamp, session, task linkage
- Messages are auto-classified by type (command, question, feedback, status)
- doey-ctl can list, search, and show stats on interactions
- TUI shows interaction history (at minimum in task detail — "user messages about this task")
- Existing sessions work unchanged — capture is additive, never blocking
- Foundation is ready for future automation: "user asked X three times → make it automatic"
