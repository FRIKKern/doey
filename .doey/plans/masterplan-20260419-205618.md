# Plan: Discord Integration for Doey

## Goal
Add an optional Discord integration — a dedicated TUI tab and a backend outbound path — that lets a Doey instance route notifications to one destination: either a (Server + Channel) via webhook, or a User DM via a personal bot. Zero external services, zero Doey-owned infrastructure, fresh-install clean.

## Context

### User decisions (AskUserQuestion, 2026-04-19)
- **Bot model:** Per-user. Each user creates their own Discord application. No Doey-owned bot or verification gate.
- **Directionality (v1):** Outbound only (Doey → Discord). No gateway WebSocket, no slash commands, no intents.
- **Binding scope:** Exactly one destination per Doey instance.

### Architecture Decision Records

**ADR-1: Per-user bot / per-user webhook.** Users own their Discord artefacts. Doey never holds shared credentials.

**ADR-2: Transport is destination-dependent.** Two `Sender` implementations behind one interface:
- `Server+Channel` → **Incoming Webhook URL** (no bot, no intents, no OAuth). Webhook-only fields (`username`, `avatar_url`) are silently dropped when the sender is `bot_dm`.
- `User DM` → **Bot token + mutual guild + DM channel** (fully specified in ADR-7).

**ADR-3: No long-lived gateway process in v1.** Outbound-only, one-shot HTTPS per send.

**ADR-4 (revised round 3): Synchronous inline sender with brief flock ownership.** `doey-tui discord send` is a short-lived subcommand. The CLI does:
1. Parse flags. Body is read from **stdin** (never from argv) to prevent `ps auxww` leakage.
2. `MkdirAll /tmp/doey/<project>/` (lazy, idempotent).
3. Acquire exclusive `flock(2)` on `/tmp/doey/<project>/discord-rl.state.lock`. Flock is held ONLY for state read → decision → state write. The HTTPS call happens WITHOUT the lock. TOCTOU: two concurrent senders may both pass breaker+coalesce checks and each hit Discord — accepted. This bounds worst-case hook latency to roughly one network round-trip (not `N × 10s` serialization).
4. Inside the lock: read state, validate cred-hash (mismatch → zero `per_route` and `breaker_open_until`; preserve `global_pause_until`), check breaker, check coalesce bucket, check global-pause. Decide `send` | `coalesce-suppress` | `breaker-skip` | `pause-skip`. Write updated state using `.tmp + fsync + rename + fsync(dir)` ONLY on transitions that require persistence.
5. Release lock.
6. For `send`: HTTPS POST with retries (3 attempts, per-retry 4s, total 10s), honoring `Retry-After` (header OR JSON body), `X-RateLimit-Scope: global`.
7. Re-acquire lock to record result: on permanent error, append ONE `O_APPEND` line to `discord-failed.jsonl` (no `bufio.Writer`, max write 4096 bytes under PIPE_BUF guarantee). Update breaker counters (success → reset `consecutive_failures`; failure → increment, open breaker at 5).
8. Release lock. Exit 0 for success OR "expected drop" (coalesced / breaker-open / no-binding / global-paused / metadata-only-no-body). Exit 1 only for config errors (unreadable creds, malformed binding).

**Consequences:** (a) dead-TUI is not a concern — no worker to host; (b) argv never carries sensitive body; (c) RL state is **project-scoped** (see ADR-4b); (d) fork-storm is bounded by the process-fork cost itself — Go binary must dispatch the `discord send` subcommand BEFORE importing TUI packages (bubbletea, sqlite) to keep cold-start <200ms p95 on a GitHub Actions ubuntu-latest 2-core runner; if that target is missed, split to a dedicated `doey-discord` binary (listed as Phase 2 fallback); (e) `flock(2)` failure on filesystems that don't support it (some FUSE, overlay) is detected and refused with a clear message — matches the POSIX-perms guard.

**ADR-4b (new round 3): RL state is project-scoped, NOT cross-project.** `/tmp/doey/<project>/discord-rl.state`. Two projects sharing one credential CAN burst past Discord's per-bucket global limit independently; the per-project sender correctly handles global 429 responses (pauses all routes in that project), but coordination across projects is NOT attempted in v1. Listed in out-of-scope. `cred_hash` in the state file is used to detect credential rotation (zero out route buckets on mismatch), not for cross-project sharing.

**ADR-5: Secrets live in `~/.config/doey/discord.conf`, `chmod 600`.** INI format. V1 writes one stanza `[default]`; reader hard-codes lookup. Named-bindings schema retained as single-line forward-compat only. POSIX-filesystem required for `chmod 600` enforcement; non-POSIX FS → refuse to load with a docs pointer.

**ADR-6: Per-instance binding pointer file.** `<project>/.doey/discord-binding` contains one line (`default` in v1). Gitignored via `.doey/.gitignore` — a file shipped inside `.doey/` so the project root `.gitignore` is never touched. `install.sh` creates `.doey/.gitignore` with the `discord-binding` entry; `on-session-start.sh` ensures it idempotently (`grep -q ... || echo >>` to avoid append races with parallel sessions). Absence → disabled. Unknown stanza name → `Error`, not silent fallback.

**ADR-7 (revised round 3): Bot-DM wizard — 7 consistent steps with OAuth2 invite.** Narrative and walkthrough are now aligned. Error differentiation is explicit.

Steps:
1. User creates Application at `https://discord.com/developers/applications` → pastes **Application ID**.
2. User creates Bot (Application → Bot tab → Reset Token) → pastes **token**. Doey `GET /users/@me` → asserts `bot: true` (401 → "Token rejected"; `bot: false` → "That looks like a user token — refuse").
3. Doey constructs OAuth2 invite URL `https://discord.com/api/oauth2/authorize?client_id=<APP_ID>&scope=bot&permissions=0` → displays to user.
4. User opens URL in browser → selects their server → Authorizes.
5. User pastes their own Discord user id (Settings → Advanced → Developer Mode → right-click self → Copy User ID).
6. Doey verifies mutual-guild: `GET /users/@me/guilds?limit=200` (with `before=` pagination if ≥200). For each guild: `GET /guilds/{id}/members/{user_id}`. Three outcomes:
   - any 200 → pick that guild, proceed.
   - all 404, bot has zero guilds → "The bot isn't in any server yet — repeat step 4".
   - all 404, bot has ≥1 guild → "The bot is in servers, but your user id ({user_id}) isn't a member of any of them. Check step 5 (user id typo?) or step 4 (wrong server?)".
7. `POST /users/@me/channels {recipient_id}` → cache `dm_channel_id`, then `POST /channels/{dm_channel_id}/messages {content: "Doey bound ✓"}`. On 200 persist creds + binding.

**Rebind semantics:** `discord.Bind()` on existing creds overwrites `[default]`, purges `dm_channel_id` cache, updates `cred_hash` in RL-state (causing per_route invalidation per ADR-4 step 4). Old `.failed.jsonl` entries are preserved but tagged with prior `cred_hash`; the tab filters to current-hash by default.

`personal-server pattern` (docs-promoted simpler alternative for DM-like use): create a 1-person server, attach a webhook to a channel there — zero bot, zero OAuth, same private-stream effect as a bot DM.

**ADR-8 (revised round 3): Privacy defaults + coalescing + redaction.**

- **Default:** metadata only (title + task-id + event-kind + duration). No bodies.
- **Boss-question exception:** first 200 **bytes on a rune boundary** of the prompt body, pre-scrubbed through the redaction helper. This is the ONLY event class that ships body-content by default.
- **Precedence:** `DOEY_DISCORD_METADATA_ONLY=1` wins over `DOEY_DISCORD_INCLUDE_BODY=1`. If both set → strict metadata.
- **Responsibility split:** The *hook* decides event-kind → "include body" policy (e.g. `stop-notify.sh` passes `--body-stdin` only for Boss questions); the *CLI* is a dumb truncator + redactor (never decides by event-kind).
- **`--title` is a stable event label, never content.** Call-sites pass short fixed labels like `"TASK_DONE"`, `"Boss question"`, `"Worker crashed"`. Content (which could contain secrets) goes through stdin only. The argv-leak fix from ADR-4 holds only if this contract is respected.
- **Redact-then-truncate order:** redaction runs over the full body first, then truncation applies to the redacted output. Truncating first could leave a partial secret that redaction missed.
- **Privacy toggles live in `~/.config/doey/config.sh` (user-level), NOT `.doey/config.sh`** — prevents accidental commit of privacy opt-outs that would surprise collaborators. The `[p]` key in the tab writes to the user-level config.
- **CLI flag naming:** `--event stop|question|error|…` (event class) vs on-disk `kind=webhook|bot_dm` (transport). No overloading.

- **Coalesce key:** `sha256(event || "|" || task_id || "|" || title)`. Different task ids with identical titles do NOT coalesce. Fallback when no task id: `sha256(event || "||" || title)`.
- **Coalesce window:** 30s. Ring cap: **32 entries** in `recent_titles` (oldest evicted). If an evicted entry has `pending_flush=true` its deferred summary is **dropped silently with a debug-log line** (ring is generously sized; data-loss path is narrow and non-critical).
- **Deferred `(×N)` flush:** a coalesced burst leaves a pending count. Flushed on EITHER (a) the next send of ANY kind (the summary rides alongside) OR (b) when the next invocation's state-read sees `ts_first_coalesce + 30s ≤ now` — that invocation emits the summary as its own send before proceeding. Guarantees: a burst never stays un-sent past 30s past its first coalesce + the next hook fire. No background timer required.
- **Test-send (`[t]` key, `doey discord send-test`) bypasses coalescing** — user-initiated probes always reach Discord.

- **Redaction patterns** (non-exhaustive; more added as discovered — documented in `tui/internal/discord/redact/README.md`):
  - OpenAI-style: `sk-[A-Za-z0-9]{20,}`
  - Anthropic: `sk-ant-api03-[A-Za-z0-9_-]+`
  - GitHub: `gh[pousr]_[A-Za-z0-9]{36,}`
  - Slack: `xox[abp]-[A-Za-z0-9-]{10,}`
  - AWS access key id: `AKIA[0-9A-Z]{16}`
  - Stripe: `sk_live_[A-Za-z0-9]{24,}`
  - Discord webhook URL: `https://discord(?:app)?\.com/api/webhooks/\d+/[A-Za-z0-9_-]+`
  - Generic bearer: `Authorization:\s*Bearer\s+\S+`
  - Private key headers: `-----BEGIN [A-Z ]+PRIVATE KEY-----`
  - `password\s*[:=]\s*\S+`, `secret\s*[:=]\s*\S+`
  - Long base64: `[A-Za-z0-9+/]{48,}=?=?`

- **Redaction surfaces (all must call the helper):**
  1. Message body sent to Discord (Boss-question excerpt branch)
  2. `/tmp/doey/<project>/discord-failed.jsonl` (every field before write)
  3. `doey doctor` stdout/stderr discord section
  4. Discord tab rendering (recent activity, failure list, status line)
  5. Bind-wizard echo-back prompts (paste of webhook URL / bot token shown as `……xxxx` last-4-only)
  6. `doey-tui discord send` stderr on error
  7. `doey discord status --json` output
  8. `doey discord failures [--tail N]` output

### Auth-flow walkthrough (fresh user)

**Webhook path** (~3 min): Bind wizard → paste URL → Doey `GET {webhook_url}` (returns 200 JSON metadata, no visible channel post) → validate shape → write creds + binding. NO implicit test-send. User may press `[t]` or run `doey discord send-test` for an explicit test.

**Bot DM path** (~7 min): Follows ADR-7 steps 1-7 exactly. The OAuth2 invite URL is generated by Doey and displayed; mutual-guild verification uses paginated `GET /users/@me/guilds`; user-id vs invite errors are differentiated (ADR-7 step 6).

### Failure modes and recovery

| Failure | Detection | Handling |
|---------|-----------|----------|
| Webhook URL 401/404 | HTTP status | `Error`, append to failure log, desktop-notify once |
| Bot token revoked | `GET /users/@me` 401 | Same |
| Bot removed from mutual guild | `POST DM` 403 | Same |
| Channel deleted | 404 on send | Same |
| 429 per-route | `X-RateLimit-*` | Honor `Retry-After` until `reset_unix` |
| 429 global | `X-RateLimit-Scope: global` header OR JSON body `{"global": true, "retry_after": …}` | Set `global_pause_until`; ALL routes block until expiry |
| `Retry-After` as float in body | Parse from JSON if header absent | Honor as seconds |
| Network unreachable | connect timeout 5s | Count toward breaker |
| Breaker open (≥5 consecutive fails) | `breaker_open_until > now` | Skip send, append one `breaker-open` failure line, exit 0 fast |
| Creds file missing / binding present | `stat` on conf | `Error`, desktop-notify once per session |
| Creds parse error | INI parse | `Error` |
| Unknown stanza name | Lookup miss | `Error`, explicit message |
| `flock(2)` unsupported | flock error EINVAL/ENOTSUP | Refuse with docs link |
| Duplicate burst (same event+task+title in 30s) | hash in `recent_titles` | Coalesce, increment count, suppress send |
| Coalesce flush deferred | `ts_first_coalesce + 30s ≤ now` on next invocation | Emit summary send before proceeding |
| Payload > 2000 bytes | pre-send check | Truncate on **rune boundary** to ≤1990 bytes + `… (truncated)` |
| Embed field overflow | check per-field limits (title 256, description 4096, field value 1024, total 6000) | Truncate each on rune boundary |
| Bind wizard abort mid-flow | ctrl-c / esc | No creds write, no binding write |
| kill-mid-flock | kernel-released by `flock(2)` | Next sender proceeds normally (unit-tested) |
| `.failed.jsonl` concurrent append | `O_APPEND` + single <4096-byte write | Atomic per POSIX; no interleaving |
| `discord-rl.state` partial write | `.tmp + fsync + rename + fsync(dir)` under flock | Reader never sees torn file |

### Out of scope (v1)

- Bidirectional Discord → Doey routing (slash commands, replies, buttons).
- Multiple destinations per instance / fan-out.
- File attachments. Text + embeds only.
- Voice, threads, reactions.
- Encryption-at-rest for creds file.
- Shared Doey-owned bot application.
- `doey msg` / two-way chat.
- Windows support.
- Background retry of `.failed/` entries — user must manually re-send (labeled "Retry (may duplicate)" per ADR-4 step 6 TOCTOU disclosure).
- Non-POSIX filesystems / filesystems without `flock(2)`.
- **Cross-project rate-limit coordination** (ADR-4b) — each project maintains its own RL state.
- Hook-side pre-filter to bound fork-storm before it hits the Go binary (considered for v1.1).
- Worker-side `DOEY_DISCORD_INCLUDE_BODY=1` env-var defense (can't guard env writes; documented as trust-your-workers).

  - **Outer `timeout` wrapper:** the Go CLI enforces its own 10s total cap (ADR-4 step 6), which is the primary guarantee. An optional outer `timeout(1)` wrapper is belt-and-suspenders only. Because GNU `timeout` is absent on base macOS, the hook detects `command -v timeout || command -v gtimeout` and uses the first available; if neither present, the wrapper is skipped and we rely on the Go-internal cap alone. Stop-notify's existing async classification is preserved (the hook already runs off the critical path).

## Risks
- **Webhook URL / bot token leakage** — mitigated by: argv never carries body; redaction on 8 surfaces (ADR-8); `chmod 600` on creds; paste echoes `……xxxx`.
- **Go cold-start overruns budget** — mitigation: subcommand dispatch before TUI imports; fallback to dedicated `doey-discord` binary if CI gate fails. Target measured: p50 ≤ 200ms, p95 ≤ 400ms on `ubuntu-latest`.
- **Hook latency under fanout** — mitigation: flock held only for state read/write, NOT across HTTPS; worst-case 8 concurrent senders p95 ≤ 2s validated by test.
- **TOCTOU between flock-drop and HTTPS** — documented consequence: two senders may both send. Acceptable since (a) Discord's own dedup is weak anyway, (b) breaker + coalesce stabilize under load, (c) eliminating would require holding lock across HTTPS which is worse.
- **Fork-storm amplification** — mitigation is *inside* the Go binary (coalesce/breaker return fast without HTTP). Cost: 1000 events still forks 1000 Go procs. If this matters in practice, hook-side pre-filter lands in v1.1.
- **Cross-project RL independence** — ACK'd limitation; per-project state is simpler and correct per-project; documented in ADR-4b and out-of-scope.
- **Bot-DM wizard abandonment** — docs lead with `personal-server pattern`; Phase 3 is optional; Phase 4 wizard hides bot_dm when Phase 3 absent.
- **Rebind while `.failed.jsonl` non-empty** — old entries tagged with prior `cred_hash`; tab filters to current by default; "clear old failures" action.
- **POSIX / flock filesystem assumption** — refuse-with-docs, matches existing patterns; rare.
- **Redaction regex incompleteness** — documented non-exhaustive; README lists known gaps and encourages PRs.
- **Retry-(may-duplicate)** — explicit UI label; at-least-once contract stated in `docs/discord.md`.
- **`install.sh` / `on-session-start.sh` race appending to `.doey/.gitignore`** — `grep -q X file || echo X >> file` pattern (not atomic, but duplicate lines are benign and git doesn't care).
- **OAuth2 URL permission-scope drift** — we use `permissions=0`; tracked via Discord developer changelog; non-zero perms would be a plan amendment.
- **Breaker clear UX** — `[c]` with confirm prompt + optional one-shot probe; still imperfect if network still down.
- **Argv leak via `--title`** — titles are short and usually non-sensitive ("TASK_DONE", "Boss question"); body (the leak-prone field) is stdin. Documented caveat.
