# Smart SQLite search â€” URL extraction table + FTS5 full-text across tasks/messages

## Goal

User verbatim:
> "We need to be able to search in the SQLITE database really smart.. like if we for example added a figma link long ago for a task - we should be able to find it - any ideas how we can do this with Doey?"

Ship a smart search across the Doey SQLite DB so a question like "where did I paste that Figma link 3 weeks ago?" gets a sub-second answer.

Two layers in scope:
- **Layer 1 â€” URL extraction table.** Every URL written into any task field (title / description / report / decision / note / subtask / comment) is regex-extracted on write/update into `task_urls(task_id, url, host, kind, field, ts)`. Host-based `kind` tagging (figma / github / slack / linear / sanity / loom / notion / generic). CLI: `doey search --url figma`, `doey search --url 'figma.*Header'`.
- **Layer 2 â€” FTS5 full-text search.** SQLite FTS5 virtual table mirroring the searchable text fields of every task. Triggers keep it in sync. BM25 ranking, snippet/highlight. CLI: `doey search "auth token"`, `doey search --type decision "stagehand"`.

Layer 3 (semantic embeddings via sqlite-vec) is **deferred** to a future plan. This plan ships layers 1+2 only.

Surfaces in scope:
1. `doey search â€¦` CLI subcommand (router + flags + output formats).
2. Go TUI search bar in task list pane (live filter, keybind `/`).
3. `mcp__doey__search` MCP tool â€” Claude can search the DB during tasks (`url_search`, `text_search` tools).
4. `doey msg search` variant for the message DB (separate FTS5 table, same query path).
5. Backfill â€” index every existing task and message on first run, idempotently.

## Steps

- [ ] **Step 1: Schema + migration** (complexity: medium) â€” Add `task_urls` table, `tasks_fts` FTS5 virtual table, `messages_fts` FTS5 virtual table. Migration is idempotent (CREATE TABLE IF NOT EXISTS, CREATE VIRTUAL TABLE IF NOT EXISTS). Bump schema version. Apply on `doey` startup if needed.
- [ ] **Step 2: URL extractor** (complexity: low) â€” Regex set for URLs (RFC 3986-ish). Host classifier (regex-mapped to `kind`). Single function `extract_and_store_urls(task_id, field, content)` called from any field-write codepath. Idempotent on re-run (DELETE rows for `(task_id, field)` then INSERT new set).
- [ ] **Step 3: FTS5 triggers** (complexity: medium) â€” INSERT/UPDATE/DELETE triggers on `tasks` (and the related sub-tables: reports, decisions, notes, subtasks) that keep `tasks_fts` in sync. Concatenate fields into a single rowid-keyed text blob for ranking. Same for `messages` â†’ `messages_fts`.
- [ ] **Step 4: Backfill** (complexity: low) â€” One-shot pass on first run after migration: iterate every task and message, run extractor + FTS insert. Progress bar in TUI. Rerunnable safely (idempotent).
- [ ] **Step 5: `doey search` CLI** (complexity: medium) â€” New subcommand in `shell/doey.sh` routing to `tui/cmd/search` (Go) for actual query execution against the DB. Flags: `--url <pattern>`, `--type <field|task_type>`, `--limit N`, `--json`, `--since <date>`. Output: ranked list with task ID, title, snippet, matched-URL highlight. Exit 0 found, 1 none.
- [ ] **Step 6: Go TUI search bar** (complexity: medium) â€” Add `/` keybind in task list pane to focus a search input. Live filter as user types (debounced). Shows match count + scrollable result list. Enter opens task detail. Esc clears.
- [ ] **Step 7: MCP tool exposure** (complexity: medium) â€” Add `doey-search` MCP server (or extend existing doey MCP) with two tools: `url_search(pattern, kind?)` and `text_search(query, type?)`. JSON shape matches the CLI `--json`. Document in `docs/mcp-tools.md`. Wire so Claude (any role) can use it via `mcp__doey__search` when the skill loads.
- [ ] **Step 8: `doey msg search`** (complexity: low) â€” Mirror the CLI on the message DB. Same flag shape. Different table (`messages_fts`). Useful for "find that decision msg from W2 last Tuesday."
- [ ] **Step 9: Tests** (complexity: medium) â€” Unit tests on URL extractor (40+ URL fixtures, edge cases: trailing punct, parens, query strings, fragments). Integration test: insert task with figma URL â†’ `doey search --url figma` returns it. FTS test: insert task with body "auth token expiry" â†’ `doey search "auth"` returns it ranked first. Backfill test: pre-populate DB, run migration, assert all rows indexed. Bash 3.2 + zsh-safe shell wrapper.
- [ ] **Step 10: Docs + UX polish** (complexity: low) â€” `docs/search.md` with examples, query syntax, supported URL kinds, FTS5 query operators (AND/OR/NOT/NEAR/quoted phrases). Add to README + `doey --help`.

## Risks

- **FTS5 trigger fragility on schema changes** â†’ keep the trigger code in one file (`tui/internal/store/migrations/fts.go`), regenerate on every migration. Add a `doey doctor --fts` self-check that verifies the FTS rowcount matches the parent table.
- **URL regex over/under-matches** â†’ ship a curated set of test URLs in fixtures; use `mvdan.cc/xurls` (Go) for robust extraction rather than hand-rolled regex if available.
- **Backfill on big DBs (10k+ tasks)** â†’ run inside a transaction, batched N=500 rows per commit, progress reported via TUI status bar. Store a `last_backfilled_task_id` watermark so it can resume after interrupt.
- **MCP tool injection risk** (claude searches DB â†’ leaks task content) â†’ MCP tool only reads; results are scoped to current project DB. No cross-project search. Document the trust model.
- **Performance regression on hot insert path** â†’ URL extractor runs synchronously after the field write; benchmark to ensure < 5ms on a 10KB body. If it ever exceeds 50ms, move to async queue.
- **TUI search bar stealing keystrokes** â†’ only active when task list pane has focus AND `/` was pressed; clear pattern restores normal nav. Add an explicit "search mode" indicator in the pane border.
- **Conflict with existing `doey search` if it exists** â†’ check first; if there's any existing namespace, prefix as `doey find` or scope the new command behind `doey task search` / `doey msg search`.
- **Fresh-install invariant** â†’ migration must be a no-op on a fresh DB (CREATE TABLE IF NOT EXISTS only). Tested in `doey doctor`.
