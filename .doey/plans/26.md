# PoC Verification Instructions — prove the fix works, not just what changed

## Goal
Enhance Proof of Completion so every task includes actionable verification instructions — concrete steps to prove the fix works or the feature does what's intended. Currently PoC only shows files changed and a summary. The user needs to know HOW to verify the problem is actually solved.

## Steps
- [ ] Step 1: Research current PoC schema, emission pipeline, and stop-hook capture to understand exactly what fields exist today and where they flow (complexity: low)
- [ ] Step 2: Add verification_steps field to the proof schema — both in the PROOF emission format (stop-results.sh) and the SQLite/doey-ctl layer (complexity: medium)
- [ ] Step 3: Update worker agent prompts to require verification steps in every PROOF emission — workers must provide concrete testable instructions like "run X command and expect Y output" or "open TUI, navigate to Z, confirm W appears" (complexity: medium)
- [ ] Step 4: Update stop-results.sh to capture and persist verification_steps alongside existing proof data (complexity: low)
- [ ] Step 5: Update Go TUI task detail proof section to display verification instructions prominently — separate from the "what changed" summary, visually distinct (complexity: medium)
- [ ] Step 6: Update Task Reviewer agent to check that verification steps are present, concrete, and actionable — reject tasks with vague or missing verification (complexity: medium)
- [ ] Step 7: Build and test end-to-end — create a test task, verify worker emits verification steps, TUI displays them, reviewer gates on them (complexity: low)

## Risks
- Schema migration: adding a field to proof data must be backward-compatible with existing tasks that lack it (mitigation: default to empty/null, TUI shows "No verification steps" gracefully)
- Agent prompt bloat: adding verification requirements to worker prompts must be concise so it doesn't waste context (mitigation: 2-3 lines max, with examples)
- Verification quality: workers might write vague steps like "test it" (mitigation: Task Reviewer enforcement + examples in prompt showing good vs bad)

## Scope
- **Files**: stop-results.sh (emission), proof schema, doey-ctl proof handling, TUI task detail view (Go), worker agent template, Task Reviewer agent template
- **Teams**: 1 team, focused scope
- **Workers**: 2-3 (schema+hooks worker, TUI worker, agent prompts worker)

## Acceptance Criteria
- Every new task completion includes a verification_steps field with concrete, testable instructions
- TUI task detail shows verification steps in a visually distinct section under proof
- Task Reviewer rejects completions that lack verification steps or have vague ones
- Existing tasks without verification steps display gracefully (no crashes, shows placeholder)
- A completed task's proof section answers "how do I know this actually works?" not just "what was changed?"
