# Centralized Role Definitions

## Goal
One file defines all role names and metadata. Shell, Go, agents, and skills all derive from it. A rename = change one file + run one command.

## Design

### Source of truth: `shell/doey-roles.sh`
```bash
# Role display names (the ONLY place these are defined)
DOEY_ROLE_COORDINATOR="Taskmaster"
DOEY_ROLE_TEAM_LEAD="Subtaskmaster"
DOEY_ROLE_BOSS="Boss"
DOEY_ROLE_WORKER="Worker"
DOEY_ROLE_FREELANCER="Freelancer"
DOEY_ROLE_INFO_PANEL="Info Panel"
DOEY_ROLE_TEST_DRIVER="Test Driver"

# Role identifiers (stable, never renamed — used in code paths)
# These are the internal IDs: coordinator, team_lead, boss, worker, freelancer
```

### Consumers

1. **Shell/hooks** — `source doey-roles.sh` at top. Replace all hardcoded "Taskmaster"/"Subtaskmaster" strings with `$DOEY_ROLE_COORDINATOR` / `$DOEY_ROLE_TEAM_LEAD`
2. **Go TUI** — `go generate` script reads doey-roles.sh, writes `tui/internal/roles/generated.go` with const block
3. **Agent definitions** — Template syntax `{{ROLE_COORDINATOR}}` in `.md.tmpl` files. `install.sh` or a build step expands templates → final `.md` files
4. **Skills** — Same template approach as agents
5. **CLAUDE.md** — Manual (changes once per rename, acceptable)
6. **Docs** — Manual or templated (low priority)

### Rename workflow after this
1. Edit `shell/doey-roles.sh` (one file)
2. Run `doey build` or `make generate` — regenerates Go constants + expands agent/skill templates
3. Commit

## Risks
- Template expansion adds a build step — fresh installs must run it (install.sh handles this)
- go:generate requires the shell script to be accessible at generate time
- Agent templates add complexity — if too heavy, skip and keep agents as authored files

## Acceptance Criteria
- Role names defined in exactly ONE file
- Shell scripts use variables, not string literals for roles
- Go TUI uses generated constants
- Agent/skill templates expand from source of truth
- A rename = edit 1 file + run 1 command
- go build + bash compat pass
